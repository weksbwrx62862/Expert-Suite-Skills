# 常见错误对照表

基于现有10个专家套件的分析，归纳常见错误和修正方法。

## Skill级错误

### 1. 触发词覆盖不足

**表现**：
- description中只有1-2个触发词
- 只有命令式表达，没有疑问句式
- 缺少口语化表达

**示例**：
```yaml
# 错误
description: >
  审查合同风险。
  触发词："审查合同"。

# 正确
description: >
  上传合同文件，逐条审查风险并按红黄绿分级标记。
  当用户要求"审查合同"、"合同审查"、"帮我审一下合同"、
  "这个合同能签吗"、"合同有问题吗"、"看看合同"、
  "合同风险"、"合同有没有坑"时触发此技能。
```

**修正**：补充同义表达、疑问句式、口语化说法。参见 [trigger-design-guide.md](trigger-design-guide.md)。

### 2. 连接器依赖

**表现**：
- 无连接器时Skill无法完成核心功能
- "独立能力"section为空或过于简单
- 所有功能都标注需要连接器

**示例**：
```markdown
# 错误
**独立能力（无需连接器）**
- 基础分析

**增强能力（连接器加持）**
- ~~Notion → 将报告写入Notion
- ~~数据库 → 获取财务数据
- ~~邮件 → 发送报告

# 正确
**独立能力（无需连接器）**
- 基于上传材料的结构化分析
- 完整的输出报告生成
- 质量检查和风险提示

**增强能力（连接器加持）**
- ~~Notion → 将报告写入Notion知识库
```

**修正**：确保所有核心功能在无连接器时可用。连接器只增强便利性。

### 3. 输出模板不完整

**表现**：
- 输出格式只有大纲，没有具体模板
- 占位符没有说明填充逻辑
- 缺少表格、代码块等结构化格式

**示例**：
```markdown
# 错误
## 输出格式

输出审查报告，包含风险概览、逐条审查、谈判策略。

# 正确
## 输出格式

```
## 合同审查报告

**合同名称**：[名称]
**签约双方**：甲方 [名称] / 乙方 [名称]

## 风险概览
- 红色（需谈）：[X]项
- 黄色（需改）：[Y]项

## 逐条审查

### [条款类别] -- [通过/需改/需谈]
**合同原文**：[引用]
**审查意见**：[分析]
**修改建议**：[替代条款措辞]

## 谈判策略
### 第一梯队（必须争取）
...
```
```

**修正**：提供完整的输出模板，每个占位符标注填充逻辑。

### 4. 质量标准不可验证

**表现**：
- 质量标准使用模糊词汇（"专业"、"深入"、"规范"）
- 没有具体的检查方法
- 缺少红线规则

**示例**：
```markdown
# 错误
## 质量标准

- 报告要专业
- 分析要深入
- 格式要规范

# 正确
## 质量标准

- 每项审查意见必须引用具体法条或司法解释条文
- 修改建议必须提供具体替代条款措辞，非笼统方向性建议
- 风险等级判定必须有明确理由
- 违约金审查必须按量化校验规则检查合理性
- 不得引用外国法律作为主要依据
```

**修正**：每条质量标准都使用可验证的具体标准。参见 [quality-checklist.md](quality-checklist.md)。

### 5. SKILL.md 过于冗长

**表现**：
- SKILL.md 超过500行
- 包含大量概念解释和背景说明
- 核心流程被淹没在细节中

**修正**：
- 核心流程保留在SKILL.md
- 详细规范移入references/
- 概念解释删除或简化

## 套件级错误

### 6. plugin.json 与文件夹不一致

**表现**：
- `skills` 数组中的名称与文件夹名称不一致
- 缺少 `displayName` 或 `description`
- `name` 不符合 kebab-case

**示例**：
```json
// 错误
{
  "skills": ["深度报告", "行业研究"]
}
// 文件夹实际是：skills/深度研报/、skills/行业分析/

// 正确
{
  "skills": ["深度研报", "行业分析"]
}
```

**修正**：逐项比对 `skills` 数组与文件夹名称，确保完全一致。

### 7. README.md 缺少关键信息

**表现**：
- 没有能力层级图
- 没有Skill工作流图
- 没有适用角色说明
- 没有免责声明

**修正**：按照 [readme-template.md](readme-template.md) 的模板补充。

### 8. 版本号不规范

**表现**：
- 版本号使用 "v1.0" 或 "1.0"（缺少修订号）
- 版本号使用日期格式 "2026-04-28"
- 小修改却升级主版本号

**修正**：使用 semver 格式 `X.Y.Z`，参见 [plugin-json-spec.md](plugin-json-spec.md)。

## References 使用错误

### 9. 核心内容放入references

**表现**：
- SKILL.md 中只有目录，所有内容在references/
- 执行流程在references/中
- 触发条件在references/中

**修正**：核心工作流程、触发条件、输出模板、质量标准必须放在SKILL.md。详细规范、市场数据、长篇幅文档放references/。

### 10. references 文件没有引用

**表现**：
- references/ 文件夹中有文件，但SKILL.md中没有引用
- 引用路径错误
- 引用了不存在的文件

**修正**：在SKILL.md中使用 `[文件名](references/文件名.md)` 格式引用，确保文件存在。

## 跨Skill一致性错误

### 11. 术语不一致

**表现**：
- 同一概念在不同Skill中用不同名称
- 连接器占位符写法不一致（有的用 `~~Notion`，有的用 `~~notion`）
- 输出格式风格不统一

**修正**：建立术语表，统一用词。所有Skill使用相同的占位符格式。

### 12. Skill之间无关联

**表现**：
- 套件内Skill各自独立，没有工作流串联
- 没有"关联Skill"section
- 用户不知道何时该用哪个Skill

**修正**：在README.md中绘制Skill工作流图，在每个Skill的"关联Skill"section中说明衔接关系。
