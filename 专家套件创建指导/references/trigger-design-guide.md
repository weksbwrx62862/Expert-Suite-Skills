# 触发词设计指南

触发词是用户说什么话时激活Skill。这是Skill的入口，设计好坏直接决定Skill能否被正确调用。

## 核心规则

1. **覆盖同义表达**：同一意图至少有3种不同说法
2. **包含疑问句式**：用户常以问题形式提出需求
3. **避免过度宽泛**：一个词不能触发所有场景
4. **中英文都写**：description和description_en都要包含

## 覆盖维度检查表

| 维度 | 说明 | 示例（审查合同） |
|------|------|----------------|
| **直接命令** | 动词+名词 | "审查合同"、"合同审查" |
| **请求帮助** | "帮我/给我/帮我看看" | "帮我审一下合同" |
| **疑问句式** | "能/吗/怎么样" | "这个合同能签吗"、"合同有问题吗" |
| **口语化表达** | 非正式说法 | "看看合同"、"合同有没有坑" |
| **英文表达** | description_en中 | "review a contract", "contract risks" |

## 触发词数量标准

| Skill复杂度 | 最少触发词数 | 示例 |
|------------|-------------|------|
| 简单Skill（单一功能） | 5个 | 简历筛选 |
| 中等Skill（2-3个分支） | 8个 | 营销文案 |
| 复杂Skill（多类型分支） | 10个+ | 审查合同、法律文书 |

## 反模式

| 反模式 | 错误示例 | 正确做法 |
|--------|---------|---------|
| **过于宽泛** | "帮我处理文档"（什么文档？） | "审查合同"、"起草合同"分别触发不同Skill |
| **过于狭窄** | 只有"深度报告"一个词 | 加上"公司深度"、"首次覆盖"、"写一篇研报" |
| **只有命令式** | "审查合同"、"合同审查" | 加上"帮我审一下"、"这个合同有问题吗" |
| **忽略英文** | description_en为空 | 英文触发词与中文一一对应 |
| **包含内部术语** | "调用合同审查Skill" | 用户不会说"Skill"，用自然语言 |

## 验证方法

写完description后，用以下问题自检：

- [ ] 列出至少3个同义词或近义表达
- [ ] 包含至少1个疑问句式
- [ ] 包含至少1个口语化表达
- [ ] 英文description包含对应触发词
- [ ] 没有使用用户不会说的内部术语

## 优秀示例

### 审查合同

```yaml
description: >
  上传合同文件，逐条审查风险并按红黄绿分级标记，输出修改建议和三梯队谈判策略。
  当用户要求"审查合同"、"合同审查"、"帮我审一下合同"、"这个合同能签吗"、
  "合同有问题吗"、"看看合同"、"合同风险"、"合同有没有坑"时触发此技能。
```

**分析**：
- 直接命令：审查合同、合同审查
- 请求帮助：帮我审一下合同
- 疑问句式：这个合同能签吗、合同有问题吗、合同有没有坑
- 口语化：看看合同、合同风险
- 共8个触发词，覆盖充分

### 营销文案

```yaml
description: >
  输入产品/活动信息和目标平台，生成适配的营销文案。
  当用户要求"写文案"、"公众号文章"、"小红书文案"、"营销文案"、"写推文"、
  "种草笔记"、"品牌软文"、"朋友圈文案"、"产品介绍"、"推广文案"、
  "视频号文案"、"知乎回答"、"带货文案"、"软文"、"内容创作"时触发。
```

**分析**：
- 覆盖7个平台的具体说法
- 包含"写"、"做"、"生成"等不同动词
- 包含平台名+内容类型的组合
- 共15个触发词，非常充分

## 快速测试

写完description后，模拟以下场景：

1. 用户说"帮我看看这个合同" → 应该触发
2. 用户说"这个合同能签吗" → 应该触发
3. 用户说"写一份合同" → **不应该触发**（这是"起草合同"Skill的触发词）
4. 用户说"帮我处理一下文件" → **不应该触发**（过于宽泛）

如果3和4被错误触发，说明触发词设计有问题，需要调整。
