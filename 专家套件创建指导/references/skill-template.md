# SKILL.md 标准模板

复制此模板创建新的Skill。

```markdown
---
name: Skill名称
description: >
  一句话说明Skill的核心功能和价值。
  触发词："关键词1"、"关键词2"、"关键词3"。
argument-hint: "输入提示，例如：上传文件或描述需求"
name_en: "english-name"
description_en: >
  One-sentence description of the skill's core function.
  Trigger phrases: "phrase1", "phrase2", "phrase3".
---

# /Skill名称 -- 功能标题

> 如遇到不熟悉的占位符，参见 [CONNECTORS.md](../../CONNECTORS.md)。

**独立能力（无需连接器）**

- 能力1的简要说明
- 能力2的简要说明
- 能力3的简要说明

**增强能力（连接器加持）**

- ~~ConnectorName → 连接后的增强效果

## 连接器（可选增强）

| 连接器 | 增强能力 |
|--------|---------|
| **~~ConnectorName** | 增强能力描述 |

> 没有连接器也完全可以使用——基于内置知识完成核心功能，输出以对话形式呈现。

## 调用方式

```
/Skill名称 <参数描述>
```

处理以下内容：@$1

接受文件上传（PDF、DOCX、图片）或直接粘贴文本。

## 输入要求

1. **必填项1**：说明
2. **必填项2**：说明
3. **选填项1**（可选）：说明
4. **选填项2**（可选）：说明

用户只提供部分信息也可继续，对缺失信息做合理假设并标注。

## 执行流程

### 第一步：明确需求与补全信息

- 如果信息完整：直接进入第二步
- 如果信息缺失：追问关键信息，给出示例引导

### 第二步：核心处理步骤

具体处理逻辑：
- 子步骤1
- 子步骤2
- 子步骤3

**分支处理**（如适用）：

| 条件 | 处理方式 |
|------|---------|
| 条件A | 处理方式A |
| 条件B | 处理方式B |

### 第三步：质量检查

按以下清单逐项检查：
- [ ] 检查项1
- [ ] 检查项2
- [ ] 检查项3

### 第四步：输出结果

**如已连接 ~~ConnectorName：**
- 执行增强操作

**如未连接 ~~ConnectorName：**
- 以对话形式输出结果

```

## 模板填写说明

### Frontmatter 字段

| 字段 | 必填 | 说明 | 验证方法 |
|------|------|------|---------|
| `name` | 是 | Skill中文名，与文件夹名称一致 | 检查文件夹名是否匹配 |
| `description` | 是 | 包含功能说明和触发词，这是Skill的触发机制 | 列出至少3个同义表达；包含疑问句式 |
| `argument-hint` | 否 | 参数输入提示，帮助用户理解如何调用 | 检查是否包含输入示例 |
| `name_en` | 否 | 英文标识名 | 使用kebab-case |
| `description_en` | 否 | 英文描述和触发词 | 包含英文触发词 |

**Frontmatter 规则**（来自skill-creator最佳实践）：
- 只包含 `name` 和 `description` 两个字段（YAML中）
- `description` 是主要触发机制，必须包含：
  - Skill的功能说明
  - 具体的触发词/触发场景
  - 所有"when to use"信息必须放在这里，不要放在body中
- Body只在Skill触发后加载，所以"When to Use This Skill" section放在body中是无效的

### Body 结构

| 章节 | 必填 | 说明 | 验证方法 |
|------|------|------|---------|
| 能力概览 | 是 | 独立能力 + 增强能力，让用户快速了解价值 | 检查独立能力是否>=3项 |
| 连接器 | 是 | 即使无连接器也要说明"无连接器时的工作方式" | 检查是否有"> 没有连接器也完全可以使用"说明 |
| 调用方式 | 是 | 命令格式和参数说明 | 检查是否有`/命令 <参数>`格式 |
| 输入要求 | 是 | 必填/选填信息清单 | 检查是否区分了必填和选填 |
| 执行流程 | 是 | 分步骤的处理逻辑 | 检查每步是否有明确输入和输出 |
| 输出格式 | 是 | 完整的输出模板 | 检查所有占位符是否有填充逻辑说明 |
| 质量标准 | 是 | 可验证的质量要求 | 检查是否使用具体标准而非模糊词汇 |
| 关联Skill | 否 | 套件内Skill的串联关系 | 检查是否说明了衔接场景 |

### 触发词设计

触发词是用户说什么话时激活这个Skill。设计要点：

- **覆盖同义表达**：如"审查合同"、"合同审查"、"审合同"
- **包含疑问句式**：如"这个合同能签吗"、"合同有问题吗"
- **避免过度宽泛**：不要用一个词触发所有场景
- **中英文都写**：description和description_en都要包含

详细设计标准参见 [trigger-design-guide.md](trigger-design-guide.md)。

### 质量标准的写法

好的质量标准是可验证的：

| 差示例 | 好示例 | 验证方法 |
|--------|--------|---------|
| "输出要专业" | "法条引用必须精确到条款项" | 检查输出中每条法条是否精确到条/款/项 |
| "格式要规范" | "金额必须同时使用大小写" | 检查金额是否同时有大写和小写 |
| "分析要深入" | "ROE分析须用杜邦分解" | 检查ROE分析是否分解为净利率×资产周转率×权益乘数 |

### References 使用

| 放SKILL.md | 放references/ | 原因 |
|-----------|--------------|------|
| 核心工作流程 | 详细格式规范 | 核心流程必须触发即加载 |
| 触发条件 | 市场数据区间 | 触发条件决定Skill是否激活 |
| 输出模板 | 长篇幅参考文档 | 模板是输出的一部分 |
| 质量标准 | 脚本和模板文件 | 标准是核心要求 |

**Progressive Disclosure 原则**：
- Level 1（始终加载）：Metadata（name + description）~100词
- Level 2（触发后加载）：SKILL.md body < 500行
- Level 3（按需加载）：References 无限制

保持SKILL.md body在500行以内。超过时拆分内容到references/。
