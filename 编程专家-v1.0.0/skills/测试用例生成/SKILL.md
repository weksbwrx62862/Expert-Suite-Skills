# /测试用例生成 -- 智能测试用例生成

> 如遇到不熟悉的占位符，参见 [CONNECTORS.md](../../CONNECTORS.md)。

**独立能力（无需连接器）**

- 分析代码逻辑生成测试用例
- 覆盖正常路径、边界条件和异常路径
- 生成符合主流测试框架的测试代码
- 计算测试覆盖率预估

**增强能力（连接器加持）**

- ~~Filesystem → 读取项目测试配置，生成一致的测试代码

## 连接器（可选增强）

| 连接器 | 增强能力 |
|--------|---------|
| **~~Filesystem** | 读取项目现有测试代码，保持测试风格和框架一致 |

> 没有连接器也完全可以使用——基于通用测试最佳实践生成。

## 调用方式

```
/测试用例生成 <被测代码>
```

为以下代码生成测试：@$1

## 输入要求

1. **被测代码**（必填）：需要测试的函数/类/模块
2. **测试框架**（必填）：如Jest、Pytest、JUnit、Go testing
3. **编程语言**（必填）：代码使用的语言
4. **测试类型**（可选）：单元测试/集成测试/全部，默认全部
5. **已有测试**（可选）：如果已有部分测试需要补充

## 执行流程

### 第一步：Spec场景覆盖检查（Harness增强）

生成测试前先检查spec中的scenario：
- **如有spec**：确保每个scenario至少有一个对应的测试用例
- **spec → 测试映射**：
  ```
  Scenario: [scenario名称] → Test: [测试用例名称]
  ```
- **无spec**：基于代码逻辑生成测试，但提示建议补充spec

### 第二步：代码分析

- 识别被测函数的输入参数和返回值
- 识别条件分支（if/else、switch、try-catch）
- 识别外部依赖（数据库、API、文件系统）
- 识别边界条件（空值、空数组、最大值、最小值）

### 第三步：生成测试场景

| 场景类型 | 说明 | 示例 |
|---------|------|------|
| **正常路径** | 标准输入，预期成功 | 正常参数调用 |
| **边界条件** | 临界值输入 | 空数组、最大长度、0值 |
| **异常路径** | 错误输入，预期失败 | null、负数、类型错误 |
| **状态相关** | 依赖对象状态的测试 | 对象初始化后/操作后 |
| **并发相关** | 多线程/协程场景 | 同时读写、竞态条件 |

### 第四步：编写测试代码

- 使用 Arrange-Act-Assert 结构
- 为外部依赖创建Mock/Stub
- 使用描述性的测试名称
- 每个测试只验证一个概念

### 第五步：覆盖率分析

- 语句覆盖率预估
- 分支覆盖率预估
- 未覆盖路径说明
- **Spec覆盖检查**：确认所有scenario都有对应测试

## 质量标准

- 每个测试必须有明确的Arrange-Act-Assert结构
- 测试名称必须描述行为和预期结果（如`shouldReturnErrorWhenInputIsNull`）
- 边界条件必须覆盖：空值、空集合、最大值、最小值、边界值+1/-1
- 异常测试必须验证具体的异常类型和错误消息
- 外部依赖（数据库、API、文件）必须使用Mock，不能真实调用
- 不得生成无意义的测试（如测试getter/setter且不做断言）
