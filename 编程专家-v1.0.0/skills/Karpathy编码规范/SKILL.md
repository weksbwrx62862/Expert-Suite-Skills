# /Karpathy编码规范 -- Karpathy编码哲学

基于Andrej Karpathy对LLM编码陷阱的观察，提供4条核心编码原则。参见 [karpathy-examples.md](../../references/karpathy-examples.md) 获取完整实战案例。

**独立能力（无需连接器）**

- 解释4条核心编码原则
- 评估代码是否符合Karpathy规范
- 提供具体改进建议
- 生成符合规范的代码

## 调用方式

```
/Karpathy编码规范 <代码或问题>
```

评估以下代码：@$1

## 输入要求

1. **代码片段**（可选）：需要评估的代码
2. **具体问题**（可选）：如"这段代码是否过度设计"
3. **场景描述**（可选）：编码场景说明

## 四大原则

### 原则1：编码前思考（Think Before Coding）

**不要假设。不要隐藏困惑。暴露权衡。**

编码前必须：
- 明确陈述假设——如果不确定，提问而不是猜测
- 存在多种理解时，列出选项——不要默默选择
- 存在更简单方案时，说出来——该推回时推回
- 遇到不清楚的地方，停下来——说出困惑所在，提问

**反模式**：
- 用户说"导出用户数据"，直接写导出所有用户的代码（假设了范围、格式、字段）
- 用户说"让搜索更快"，直接加缓存和索引（假设了"更快"的含义）

**正确做法**：
```
导出用户数据前，我需要澄清：
1. 范围：导出全部用户还是筛选后的子集？
2. 格式：JSON/CSV/直接API返回？
3. 字段：哪些字段？（有些可能涉及隐私）
```

### 原则2：简洁优先（Simplicity First）

**最少代码解决问题。不要 speculative。**

- 不要超出需求的功能
- 不要为单次使用代码做抽象
- 不要未被要求的"灵活性"或"可配置性"
- 不要为不可能场景做错误处理
- 写了200行但50行能搞定，重写

**自检问题**："资深工程师会说这过度复杂吗？"如果是，简化。

**反模式**：
- 用户说"计算折扣"，生成Strategy模式+抽象类+配置对象（30行设置做简单计算）
- 用户说"保存用户偏好"，加缓存、验证、合并、通知（没人要求的功能）

**正确做法**：
```python
# 简单直接
def calculate_discount(amount: float, percent: float) -> float:
    return amount * (percent / 100)
```

### 原则3：精准修改（Surgical Changes）

**只碰必须碰的。只清理自己制造的。**

编辑现有代码时：
- 不要"改进"相邻代码、注释或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你自己会不同做法
- 发现无关死代码，提一下——但不要删

你的修改制造孤儿时：
- 删除**你的修改**导致的未使用import/变量/函数
- 不要删除预先存在的死代码，除非被要求

**测试标准**：每一行修改都能追溯到用户的请求。

**反模式**：
- 修复空邮箱Bug时，顺便"改进"邮箱验证、加用户名验证、改注释、加docstring
- 添加上传日志时，改引号风格、加类型提示、重排空白

**正确做法**：
```diff
-     if not user_data.get('email'):
+     email = user_data.get('email', '')
+     if not email or not email.strip():
          raise ValueError("Email required")
```

### 原则4：目标驱动执行（Goal-Driven Execution）

**定义成功标准。循环直到验证。**

把指令式任务转化为可验证目标：
- "加验证" → "写无效输入的测试，然后让它们通过"
- "修Bug" → "写复现Bug的测试，然后让它通过"
- "重构X" → "确保重构前后测试都通过"

多步骤任务时，陈述简要计划：
```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

强成功标准让你独立循环。弱标准（"让它工作"）需要持续澄清。

## 质量标准

- 评估必须基于代码实际内容，不能泛泛而谈
- 每个原则的评估必须有具体的代码位置引用
- 改进建议必须提供具体的修改前后对比
- 不得为符合规范的代码强行找问题
- 评估应平衡：不是追求绝对完美，而是避免常见陷阱
