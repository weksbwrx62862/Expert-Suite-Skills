# /代码生成 -- 智能代码生成

> 如遇到不熟悉的占位符，参见 [CONNECTORS.md](../../CONNECTORS.md)。

**独立能力（无需连接器）**

- 根据需求生成完整函数/类/模块
- 自动处理错误和边界条件
- 生成符合语言惯用法的代码
- 附带使用示例和说明

**增强能力（连接器加持）**

- ~~Filesystem → 读取项目现有代码，生成风格一致的代码

## 连接器（可选增强）

| 连接器 | 增强能力 |
|--------|---------|
| **~~Filesystem** | 读取项目现有代码风格，生成一致的代码 |

> 没有连接器也完全可以使用——基于通用最佳实践生成代码。

## 调用方式

```
/代码生成 <需求描述>
```

生成以下功能的代码：@$1

## 输入要求

1. **功能需求**（必填）：需要实现的功能描述
2. **编程语言**（必填）：如Python、JavaScript、Go、Java
3. **框架/库**（可选）：如React、Spring Boot、FastAPI
4. **约束条件**（可选）：如"时间复杂度O(n)"、"不使用第三方库"
5. **输入输出示例**（可选）：期望的输入和输出格式

## 执行流程

### 第一步：Spec检查（Harness增强）

编码前检查是否有明确的spec：
- **如有spec**：提取spec中的scenario作为编码验收依据，严格按Given/When/Then实现
- **如无spec**：提示用户先使用 `/Spec驱动开发` 对齐需求，或生成轻量级spec

```
✓ Spec检查: [检测到/未检测到] 需求规格
[如检测到，列出相关scenario]
```

### 第二步：编码前思考（Think Before Coding）

生成代码前，先明确陈述假设：
- 如果需求有歧义，列出多种理解并选择最简方案
- 如果存在约束未明确，说明假设的约束条件
- 如果存在更简单的方案，优先选择简单方案

**示例**：
```
假设：用户只需要基础折扣计算，不需要多种折扣策略组合。
假设：输入金额和折扣百分比都是正数。
选择：使用简单函数而非Strategy模式（当前需求单一）。
```

### 第三步：需求分析

- 理解功能需求和业务场景
- 识别核心算法或设计模式
- 明确约束条件（性能、兼容性、依赖）
- **Karpathy约束**：识别并排除超出需求的功能
- **加载项目规范**：从项目记忆中加载相关编码规范（命名约定、错误处理模式等）

### 第四步：设计方案

- 确定函数/类/模块的接口
- 设计数据结构
- 规划错误处理策略
- **Karpathy约束**：最小设计，不预先为"未来可能"做抽象

### 第五步：代码实现（Simplicity First）

生成包含以下要素的代码：
- 清晰的函数/类签名
- 输入验证和参数检查
- 核心逻辑实现（最少代码解决问题）
- 错误处理（只处理可能发生的场景）
- 边界条件处理（空值、越界、超时）
- **简洁优先**（Karpathy原则）：选择最简单的实现方式

**Karpathy约束**：
- 不要超出需求的功能
- 不要为单次使用代码做抽象
- 不要未被要求的"灵活性"或"可配置性"
- 不要为不可能场景做错误处理
- 如果200行可以50行搞定，重写

### 第六步：代码审查

- 检查代码逻辑是否正确
- 确认是否有更好的实现方式
- 验证边界条件处理
- **精准修改**（Karpathy原则）：只修改必要部分，不引入无关变更

### 第七步：补充说明（Goal-Driven）

- 使用示例
- 复杂度分析
- 注意事项
- **成功标准**：说明如何验证这段代码正确工作

### 第八步：Task Summary输出（Harness增强）

代码生成完成后输出task summary：

```
## Task Summary: 代码生成

**完成状态**: [完成/部分完成]
**生成文件**: [文件列表]
**关键决策**: [实现中做出的技术选择]
**偏差说明**: [与原始需求的差异及原因]
**验证建议**: [如何验证这段代码]
```

## 质量标准

- 生成的代码必须可直接运行，不能是伪代码
- 必须处理常见的边界条件（空输入、越界、类型错误）
- 必须包含错误处理，不能假设输入总是合法
- 变量和函数命名必须符合语言惯例（如Python用snake_case，JS用camelCase）
- 复杂逻辑必须添加注释说明
- 不得生成包含安全漏洞的代码（如SQL拼接、eval执行用户输入）
- **Karpathy红线**：不得为单一需求生成抽象类、接口、配置对象等过度设计
