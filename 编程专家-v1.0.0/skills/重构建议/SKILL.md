# /重构建议 -- 智能重构建议

> 如遇到不熟悉的占位符，参见 [CONNECTORS.md](../../CONNECTORS.md)。

**独立能力（无需连接器）**

- 识别代码坏味道（Code Smells）
- 提供具体重构步骤（Before/After对比）
- 评估重构收益和风险
- 生成重构后的完整代码

**增强能力（连接器加持）**

- ~~Filesystem → 分析整个项目的代码结构，识别跨文件重复和依赖问题

## 连接器（可选增强）

| 连接器 | 增强能力 |
|--------|---------|
| **~~Filesystem** | 分析项目整体结构，识别跨文件重复代码和循环依赖 |

> 没有连接器也完全可以使用——基于用户提供的代码片段分析。

## 调用方式

```
/重构建议 <代码片段>
```

分析以下代码的重构空间：@$1

## 输入要求

1. **代码片段**（必填）：需要重构的代码
2. **重构目标**（可选）：如"提高可读性"、"降低复杂度"、"便于测试"
3. **约束条件**（可选）：如"不能改变接口"、"保持向后兼容"

## 执行流程

### 第一步：Spec范围确认（Harness增强）

重构前确认spec范围：
- **如有spec**：确认重构是否影响spec中的scenario，确保重构后scenario仍然通过
- **变更范围**：明确重构影响哪些spec requirement，是否需要更新spec
- **无spec**：重构后建议生成delta spec记录变更

```
✓ Spec范围确认:
- 影响Scenario: [scenario列表]
- 是否需要更新spec: [是/否]
- 重构后验收标准: [标准]
```

### 第二步：识别坏味道

| 坏味道 | 特征 | 重构手法 |
|--------|------|---------|
| **过长函数** | 函数超过50行 | 提取函数 |
| **过大类** | 类承担过多职责 | 提取类 |
| **重复代码** | 相同/相似代码出现多次 | 提取公共函数 |
| **过长参数列表** | 参数超过4个 | 引入参数对象 |
| **发散式变化** | 一个类因多种原因修改 | 拆分职责 |
| **霰弹式修改** | 修改一处需要改多个类 | 搬移函数/内联 |
| **特性依恋** | 函数频繁访问其他类数据 | 搬移函数 |
| **数据泥团** | 多处出现相同数据组合 | 提取对象 |
| **Switch语句** | 大量条件分支 | 多态替代 |
| **临时字段** | 对象某些字段只在特定场景使用 | 提取类 |
| **过度设计** | 抽象类/接口/配置对象做简单计算 | **直接删除，回归简单** |
| **推测性抽象** | 为"未来可能"的需求做的抽象 | **删除，需要时再创建** |

### 第三步：评估重构价值（Simplicity First优先）

对每个识别出的坏味道：
- **影响范围**：影响多少代码/模块
- **修复成本**：重构所需工作量
- **收益**：可读性、可维护性、可测试性提升
- **风险**：是否可能引入Bug

**Karpathy优先级规则**：
1. **简化优先**：删除过度设计和推测性抽象的收益 > 重组代码结构
2. **删除 > 重组**：对于过度设计，直接删除回归简单，比重构成"更好的设计"优先级更高
3. **评估标准**：重构后代码行数应该减少或持平，不应显著增加

### 第四步：制定重构计划

按优先级排序：
1. 高收益低风险的优先执行
2. 提供Before/After代码对比
3. 说明每一步重构的具体操作
- **简洁优先**（Karpathy原则）：重构后的代码应比原来更简单

### 第五步：生成重构后代码

- 提供完整的重构后代码
- 确保代码功能不变
- 说明重构后的改进点
- **精准修改**（Karpathy原则）：每次只做一个重构，不混合多种变更

### 第六步：验证

- 检查重构是否引入新问题
- 确认性能影响
- 确保测试仍然通过

### 第七步：Delta Spec输出（Harness增强）

重构完成后输出delta spec：

```markdown
## 重构Delta Spec

### MODIFIED Requirements
- Requirement: [受影响的requirement]
  - 变更: [具体变更]
  - 影响: [对scenario的影响]

### Design变更
- [技术决策变更]

### File Changes
- `[文件路径]` (modified)
```

## 质量标准

- 重构必须保持功能不变，不能改变原有行为
- 每个坏味道必须引用Martin Fowler《重构》中的标准名称
- 重构步骤必须具体到可操作（如"将第10-25行提取为calculateTotal函数"）
- 必须评估重构风险，高风险重构必须提供回滚方案
- 重构后代码必须通过原始测试用例（如果提供）
- 不得建议过度设计（如为简单脚本引入复杂架构）
- **Karpathy红线**：简化是最高优先级的重构目标。删除过度设计比重组成"更好的设计"优先级更高
- **Karpathy红线**：重构后代码行数应减少或持平，不应显著增加
