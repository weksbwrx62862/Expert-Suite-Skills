# /代码审查 -- 智能代码审查

> 如遇到不熟悉的占位符，参见 [CONNECTORS.md](../../CONNECTORS.md)。

**独立能力（无需连接器）**

- 静态代码分析（Bug、安全、性能、风格）
- 按严重程度分级（致命/严重/警告/建议）
- 给出具体修复代码示例
- 代码异味识别与重构建议

**增强能力（连接器加持）**

- ~~Filesystem → 批量审查整个项目代码库

## 连接器（可选增强）

| 连接器 | 增强能力 |
|--------|---------|
| **~~Filesystem** | 直接读取项目文件，批量分析代码库，跨文件依赖检查 |

> 没有连接器也完全可以使用——基于用户粘贴的代码片段进行分析。

## 调用方式

```
/代码审查 <代码片段>
```

审查以下代码：@$1

接受直接粘贴的代码、代码文件上传（.js/.ts/.py/.java/.go等）。

## 输入要求

1. **代码片段**（必填）：需要审查的代码
2. **编程语言**（必填）：代码使用的语言
3. **上下文说明**（可选）：这段代码的功能、所在的模块
4. **审查重点**（可选）：如"重点关注安全性"、"检查内存泄漏"

## 执行流程

### 第一步：Spec一致性检查（Harness增强）

审查前先检查代码是否符合spec：
- **如有spec**：逐条检查spec中的scenario是否被正确实现
- **发现不一致**：标注spec与实现的偏差，建议修正spec或代码
- **无spec**：提示建议在审查后使用 `/Spec驱动开发` 补全spec

```
✓ Spec一致性检查:
- Scenario 1: [通过/失败] — [说明]
- Scenario 2: [通过/失败] — [说明]
```

### 第二步：项目规范检查（Harness增强）

检查代码是否符合项目记忆中的规范：
- 命名约定（snake_case / camelCase / PascalCase）
- 错误处理模式（异常 vs 返回码）
- 导入排序、文件组织等风格规范
- **发现新规范**：提示记录到项目记忆

### 第三步：代码理解

- 识别编程语言和框架
- 理解代码的功能和逻辑
- 识别代码的上下文（函数、类、模块）

### 第四步：多维度审查

| 维度 | 检查内容 | 权重 |
|------|---------|------|
| **Bug风险** | 空指针、数组越界、资源未释放、并发问题 | 30% |
| **安全性** | SQL注入、XSS、敏感信息硬编码、权限绕过 | 25% |
| **性能** | 时间复杂度、内存泄漏、N+1查询、循环内IO | 20% |
| **可维护性** | 命名规范、函数长度、圈复杂度、重复代码 | 15% |
| **最佳实践** | 语言特性利用、设计模式、错误处理 | 10% |
| **Karpathy维度** | 简洁优先、精准修改、编码前思考 | 附加 |

### 第五步：Karpathy维度审查

在常规审查之外，按Karpathy编码哲学进行专项审查：

**简洁优先（Simplicity First）**：
- [ ] 是否包含超出需求的功能？
- [ ] 是否存在单次使用的抽象（Strategy模式做简单计算）？
- [ ] 是否可以用更少的代码实现（200行能否变50行）？
- [ ] 是否包含推测性功能（缓存、验证、通知等未被要求的）？

**精准修改（Surgical Changes）**（针对修改的代码）：
- [ ] 是否只修改了必要的代码？
- [ ] 是否"顺手"改进了相邻代码、注释或格式？
- [ ] 是否保持了原有代码风格？
- [ ] 每一行修改是否能追溯到需求？

**编码前思考（Think Before Coding）**：
- [ ] 代码中是否有未陈述的假设（如假设数据总是存在）？
- [ ] 是否有硬编码的"魔法值"没有说明原因？
- [ ] 函数/类设计是否考虑了多种使用场景？

Karpathy维度问题不纳入 severity 分级，单独标注为 **「Karpathy建议」**。

### 第六步：问题分级

| 级别 | 说明 | 必须修复 |
|------|------|---------|
| **致命（Critical）** | 会导致系统崩溃、数据丢失、安全漏洞 | 是 |
| **严重（Major）** | 明显Bug、性能严重下降、安全隐患 | 是 |
| **警告（Warning）** | 潜在风险、代码异味、不符合规范 | 建议 |
| **建议（Suggestion）** | 可优化点、风格问题、最佳实践 | 可选 |

### 第七步：生成修复建议

对每个问题：
- 指出具体代码位置（行号/函数名）
- 说明问题原因和影响
- 提供修复后的代码示例
- 说明修复后的收益

**Karpathy建议的修复优先级**：
- 过度设计 → 先简化再审查
- 顺手重构 → 回滚无关修改
- 隐藏假设 → 添加参数校验或文档

## 质量标准

- 每个问题必须指出具体代码位置，不能只说"这段代码"
- 修复建议必须是可运行的代码，不能是伪代码或描述
- 致命问题必须说明可能导致的具体后果（如"可能导致SQL注入，攻击者可获取全表数据"）
- 不得误报（把正确代码标记为问题），不确定时标注"疑似"
- 安全审查必须覆盖OWASP Top 10相关风险
- 性能审查必须量化影响（如"时间复杂度从O(n)变为O(n²)"）
